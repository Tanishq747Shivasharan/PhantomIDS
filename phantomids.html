// 04 — HOW IT WORKS

System Workflow

Each step happens independently of the host machine. Detection in under 200ms.

01

Attacker Fires SQLi

Using tools like sqlmap, the attacker probes the target web application with malicious payloads containing SQL keywords, UNION statements, or quote characters.

// Attack initiated from Kali Linux VM
02

Network Packet Captured

The ESP32 sensor sniffs network traffic on the shared WiFi subnet (192.168.1.x) independently using packet inspection — completely invisible to the attacker.

// ESP32 operates in promiscuous mode
03

URL Decoded & Analyzed

HTTP request parameters are URL-decoded and passed through the two-signal confidence scoring engine. Checked against 20 SQL keyword signatures and special character patterns.

// Signature engine: UNION, SELECT, DROP, OR 1=1, '--
04

Two-Signal Detection

Alert fires only when both a SQL keyword AND a quote/special character are present — reducing false positives. Single signals are logged but don't trigger alert.

// Signal 1: Keyword match + Signal 2: Quote/Special char
05

Alert on LCD + Buzzer + LED

Confirmed attack triggers the 16x2 I2C LCD display, active buzzer, and red LED indicator simultaneously. Physical alert cannot be silenced remotely.

// Detection latency: under 200ms from packet to alert
06

Log Sent to Dashboard

Attack event — IP, timestamp, payload, detection method — written to SQLite with WAL mode for high-load environments and forwarded to the centralized admin dashboard.

// Tamper-proof hardware log — physical audit evidence
